# Entity Resolution & Knowledge Base Construction

Research on matching different names/aliases to the same entity and building structured knowledge bases.

---

## [DistillER: Knowledge Distillation in Entity Resolution with Large Language Models](https://arxiv.org/abs/2602.05452)

- **Authors:** Alexandros Zeakis, George Papadakis, Dimitrios Skoutas, Manolis Koubarakis
- **Year:** 2026
- **Core method:** DistillER uses knowledge distillation to transfer entity resolution capabilities from large, expensive LLMs (teachers) to smaller, efficient models (students) without requiring gold-label annotations. The framework explores three dimensions: data selection strategies for picking informative subsets, knowledge elicitation via single- and multi-teacher setups, and distillation algorithms comparing supervised fine-tuning vs. reinforcement learning.
- **Key results:** Supervised fine-tuning on noisy labels generated by LLM teachers consistently outperforms other knowledge distillation strategies and also enables high-quality explanation generation. The distilled student models achieve significant improvements in both effectiveness and efficiency over established supervised and unsupervised ER baselines.
- **Relevance to Podders:** Podders could distill a large LLM's entity-matching knowledge into a small, fast local model for alias resolution, avoiding per-call API costs while maintaining quality. The noisy-label fine-tuning approach is practical since Podders' alias mappings are inherently imperfect.

## [Cost-Efficient RAG for Entity Matching with LLMs: A Blocking-based Exploration](https://arxiv.org/abs/2602.05708)

- **Authors:** Chuangtao Ma, Zeyu Zhang, Arijit Khan, Sebastian Schelter, Paul Groth
- **Year:** 2026
- **Core method:** CE-RAG4EM introduces a cost-efficient retrieval-augmented generation architecture for entity matching that uses blocking-based batch retrieval and generation to reduce computation. It provides a unified framework for analyzing RAG systems for entity matching, focusing on blocking-aware optimizations and retrieval granularity trade-offs.
- **Key results:** CE-RAG4EM achieves comparable or improved matching quality while substantially reducing end-to-end runtime relative to standard RAG baselines. Key configuration parameters reveal an inherent trade-off between matching performance and computational overhead, providing practical design guidance.
- **Relevance to Podders:** The blocking-based approach maps well to Podders' alias resolution: group candidate mentions by token overlap or embedding similarity first, then batch-query an LLM only for ambiguous blocks. This keeps costs manageable as the entity vocabulary grows.

## [Entity Matching using Large Language Models](https://arxiv.org/abs/2310.11244)

- **Authors:** Ralph Peeters, Aaron Steiner, Christian Bizer
- **Year:** 2023 (published at EDBT 2025)
- **Core method:** Comprehensive evaluation of generative LLMs (both hosted like GPT-4 and open-source) for entity matching in zero-shot and few-shot settings. The study compares prompt designs, in-context demonstration selection, matching rule generation, and LLM fine-tuning against traditional PLM-based matchers (BERT/RoBERTa).
- **Key results:** The best LLMs need zero or only a few training examples to match PLMs fine-tuned on thousands of examples, and they show higher robustness to out-of-distribution entities. GPT-4 can generate structured explanations for matching decisions and automatically identify error classes, helping engineers debug matching pipelines. No single best prompt exists -- it must be tuned per model/dataset.
- **Relevance to Podders:** Validates that LLMs can do entity matching with minimal labeled data, which is critical for Podders' crypto/market domain where new entities appear constantly. The error-explanation capability could help identify systematic alias-mapping failures (e.g., ticker collisions like $SOL for Solana vs. Solend).

## [On Leveraging Large Language Models for Enhancing Entity Resolution: A Cost-efficient Approach](https://arxiv.org/abs/2401.03426)

- **Authors:** Huahang Li, Longyu Feng, Shuangyin Li, Fei Hao, Chen Jason Zhang, Yuanfeng Song
- **Year:** 2024
- **Core method:** Proposes an uncertainty reduction framework that initializes possible entity cluster partitions, defines their uncertainty, and then selectively queries an LLM only on the most informative matching pairs to reduce that uncertainty. It includes error-tolerant techniques for handling LLM mistakes and a dynamic adjustment method to converge on correct partitions.
- **Key results:** The selective querying approach is both efficient and effective, dramatically reducing the number of LLM API calls needed while maintaining high resolution quality. Error-tolerance mechanisms successfully handle LLM hallucinations and incorrect match decisions.
- **Relevance to Podders:** Directly applicable to Podders' alias system -- instead of asking an LLM to verify every possible alias pair, use uncertainty-based selection to query only the most ambiguous pairs (e.g., "Is MATIC the same as POL after the rebrand?"), keeping API costs low.

## [GPTKB: Enabling LLM Knowledge Analysis via Extensive Materialization](https://arxiv.org/abs/2411.04920)

- **Authors:** Yujia Hu, Tuan-Phong Nguyen, Shrestha Ghosh, Simon Razniewski
- **Year:** 2024
- **Core method:** Proposes a methodology to comprehensively materialize an LLM's factual knowledge into a structured knowledge base through recursive querying and result consolidation. As a prototype, they build GPTKB from GPT-4o-mini, producing 101 million relational triples covering 2.9 million entities.
- **Key results:** GPTKB reveals the scope and structure of GPT-4o-mini's internalized knowledge, enabling analysis of scale, accuracy, bias, knowledge cutoff, and consistency at the same time. The materialized KB makes LLM knowledge inspectable, queryable, and comparable to traditional knowledge bases like Wikidata.
- **Relevance to Podders:** The recursive querying methodology could bootstrap Podders' entity knowledge base -- systematically extract what an LLM knows about crypto projects, their aliases, related entities, and relationships, then store it as structured triples for the alias system and market graph.

---

## Summary

These five papers collectively point toward a practical architecture for Podders' entity alias system. The core insight is that LLMs are strong entity matchers even with zero or few examples (Peeters et al.), but calling them on every mention is expensive. Two strategies mitigate cost: (1) distill a large LLM's matching ability into a small local model that runs cheaply at inference time (DistillER), or (2) use smart query selection to only ask the LLM about genuinely ambiguous pairs (Li et al., CE-RAG4EM). For bootstrapping the entity knowledge base itself, GPTKB's recursive materialization approach offers a way to systematically extract structured entity relationships from an LLM and store them in Postgres, giving Podders a starting graph of entities, aliases, and relations that can then be maintained incrementally as new market entities appear.
